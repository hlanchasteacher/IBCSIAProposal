BEFORE YOU START — READ THIS

What is the IB CS Internal Assessment?

Everything below comes directly from the official IB guide. Read it carefully — you'll answer questions before moving on.

What the IA is
The IA is a computational solution — a real, coded software product you build yourself. It brings together computational thinking and programming into one assessed task.

Your submission has three parts:
  • Documentation — a single PDF, max 2,000 words, with five sections (one per criterion). Word count must be on page 1. Code, comments, and diagrams do NOT count toward the limit.
  • Video — max 5 minutes, in mp4/avi/wmv, showing the product working and your testing strategy
  • Appendices — a PDF with your full source code. Without it, you cannot get full marks on Criterion D.
WEIGHTING & TIME
SL: 30% of final grade  ·  HL: 20%  ·  Recommended: 35 hours
The five assessment criteria
CriterionWhat's assessedMarks
A — Problem specificationDefine the problem with measurable success criteria. ~300 words.4
B — PlanningDecompose the problem and plan the development timeline. ~150 words.4
C — System overviewSystem model, algorithms, and testing strategy. ~150 words.6
D — DevelopmentThe product itself — code, techniques, testing evidence. ~1,000 words.12
E — EvaluationDid you meet the success criteria? Justified improvements. ~400 words.4
Total30
Authenticity — the rules
The work must be entirely your own. Your teacher plays an important role, but within limits:
  • Your teacher reads and gives feedback (oral or written) on one draft only
  • The version after that feedback is the final submission — it cannot be retracted
  • You cannot submit the same work for both the IA and the Extended Essay
  • The IB checks authenticity via plagiarism tools (e.g. Turnitin) and may interview students
ETHICAL REQUIREMENTS — MANDATORY
Get written consent from anyone involved in development or testing before you begin. Store all data securely. Data collected can only be used for this IA.

CRITERION A · 4 MARKS · ~300 WORDS

What problem do you want to solve?

This is the foundation of your entire IA. Everything else traces back to how well you define this section.

The Criterion A rubric
MarksWhat your response must do
1–2Outlines a problem scenario · States limited success criteria · Outlines the solution in a computational context
3–4Describes the problem in terms of measurable solution requirements · States appropriate success criteria · Explains the choice of computational context

Three key terms defined by the IB:

Problem scenario — a clear description including measurable solution requirements. Relates to the real world, another field, or a computing issue.

Success criteria — measurable outcomes that indicate the product was successfully developed. Used again in Criteria B, D, and E.

Computational context — the specific area of computing chosen for the solution.
STRONG PROBLEM SCENARIO
"My school's sports club has no system to track membership, attendance, or equipment loans. The secretary manages 80+ members via shared spreadsheets with no validation, which are frequently overwritten and produce no reports. A software solution is needed to allow the secretary to add/update member records, log attendance per session, and generate an overdue equipment report."
❌ WEAK — DO NOT WRITE LIKE THIS
"I want to make an app for my school. It will be useful and help people do things more efficiently." — No specific problem, no requirements, no context.

CRITERION A + D · PRODUCT REQUIREMENTS

What will your software do?

Before writing anything, read what counts as a valid solution — and what doesn't. This affects your marks in Criterion D.

Valid solution types
All solutions must be coded by you and demonstrate algorithmic thinking. Valid forms include:
  • OOP program or stand-alone desktop application
  • Interactive web-based application with a database
  • Computer game with original logic
  • Mobile application with original code
  • Simulation or data processing tool
  • Adding original functionality to an existing system (e.g. a plugin, a new database module)

Essential requirement: you must explicitly demonstrate and document your algorithmic thinking. Products that take existing templates with no evidence of original structural modification are not appropriate.
🚫 Explicitly NOT allowed — the IB guide lists these as inappropriate products
  • A product developed using only copied code
  • A website using a web template that predetermines its structure and layout (e.g. Wix, Squarespace, Bootstrap themes without structural modification)
  • Using exemplar products provided with software (e.g. the Northwind database in Microsoft Access)
  • A copied game without major, properly documented code modifications
  • A builder/wizard/drag-and-drop app created without code development (e.g. MIT App Inventor blocks-only, Bubble, Webflow)
  • A product that violates the ethical requirements — no consent, insecure data handling

CRITERION A · SUCCESS CRITERIA

Write your success criteria

These are the backbone of your IA. The examiner checks every one of them in Criteria B, D and E.

What the IB requires
Success criteria are measurable outcomes derived from your solution requirements that indicate the product was successfully developed. They must be testable — you should be able to demonstrate each one in your video.

The test: can you write a specific test case with specific input data and a specific expected output? If yes → measurable. If not → rewrite it.
❌ NOT MEASURABLE — these will cost you marks
"The system will be fast" · "The interface will be user-friendly" · "The app will work correctly" · "Data will be stored safely" · "Users will find it easy"
✓ MEASURABLE — specific, testable
"The user can add a new member record (name, year, contact, role) in under 4 clicks from the home screen"

"The search function returns results matching a partial name within 1 second for a database of 100+ records"

"The system prevents duplicate member IDs and displays an error message if one is detected"

"The admin can generate and export an overdue loans report listing borrower, item, and days overdue"

Aim for 5–8 criteria total. Each one should map directly to a requirement from your problem scenario.

CRITERION A · COMPUTATIONAL CONTEXT

What is your computational context?

The specific area of computing you're applying. For full marks you must explain why it fits — not just name a language.

What the IB means
The computational context is "the specific area of computing that is selected to be used in the solution."

For 3–4 marks you must explain the choice — meaning you connect your problem's specific requirements to what that area of computing enables.

Common areas: Object-Oriented Programming · Relational databases · Web applications · Mobile development · Data structures and algorithms · Networking and APIs · Computer graphics / game logic · Simulation · AI / ML
✓ EXPLAINS THE CONTEXT
"I chose OOP with a relational database because the system models distinct real-world entities — Members, Sessions, Equipment — each with unique attributes. OOP allows me to encapsulate these as classes with validation methods. SQLite provides persistent storage and SQL queries enable efficient filtering and joining of tables, which the secretary needs for searching and reporting."
❌ STATES BUT DOESN'T EXPLAIN
"I chose Python because I know it well and it's easy to use." — No connection to the problem's requirements.
Check your understanding
Two questions before you write
1. A student writes: "I chose Python because it's my strongest language and it's fast to code in." Does this meet the Criterion A requirement for the computational context?
AYes — knowing the language makes it a practical, justified choice
BNo — they must explain why this area of computing suits the specific problem, not just state a language preference
CPartially — it would get 1–2 marks but not 3–4
2. For a student tracker application, which response better explains the computational context?
A"I used a database because databases are good at storing data."
B"A relational database suits this problem because student records share structured attributes (name, year, ID) and I need to query across related tables — e.g. joining attendance and student tables — which SQL handles efficiently at scale."
C"I used Python with SQLite as it's a common combination taught in class."

CRITERION B · 4 MARKS · ~150 WORDS

How will you plan and build it?

Criterion B assesses how you decompose the problem and organise your development process.

The Criterion B rubric
MarksWhat your response must do
1–2Constructs a partial decomposition of the problem · Constructs a plan addressing some success criteria
3–4Constructs a reasonable decomposition of the problem · Constructs a plan addressing the success criteria

Decomposition — breaking the problem into smaller, manageable sub-problems or components. Diagrams are explicitly encouraged.

Reasonable decomposition — breaks into essential components that support the construction of a plan.

Your plan should include a proposed chronology covering design, development, testing, and evaluation. The IB guide explicitly mentions GANTT and AGILE-style charts as effective formats.
